# Файловая система (`src/ramfs.rs`)

Документация к файловой подсистеме Tomorrow OS. Это **RAM-only** файловая
система: диска нет, всё живёт в куче ядра. Начальный набор файлов приходит из
GRUB как multiboot2-модуль (tar-архив `initrd.tar`), разбирается один раз при
старте и дальше существует как обычный `Vec<File>` в heap. Никакой иерархии
каталогов, прав и метаданных — только плоский список «имя → байты», который
можно читать, создавать, перезаписывать и удалять в рантайме.

> Цель подсистемы — дать работающие `создание / редактирование / удаление`
> файлов без диска. Носитель initrd (GRUB → heap) выбран потому, что диск и его
> драйвер здесь не нужны: как только tar скопирован в `Vec`, исходные страницы
> модуля уже не читаются.

---

## 1. Путь данных: от GRUB до `Vec<File>`

```
 GRUB ──грузит── initrd.tar ──как multiboot2-модуль──▶ физ. память
   │
   ▼
 main.rs: разбор тега модуля (type 3)
   ├─ mod_start / mod_end  →  MOD_START / MOD_END  (глобалы)
   ├─ pmm::reserve(mod_start, mod_end)   ← чтобы PMM не отдал эти страницы
   └─ heap::HEAP.init(...)               ← куча должна быть готова ДО ramfs::init
   ▼
 ramfs::init(): парсит tar по MOD_START, копирует данные в heap
   ▼
 static mut FILES: Vec<File>   ← дальше живём только здесь
```

Ключевые точки в `main.rs`:

| Шаг | Что делает | Почему важно |
|-----|-----------|--------------|
| tag type `3` | читает `mod_start`/`mod_end` из тега модуля | это физ. адреса tar-архива в памяти |
| `pmm::reserve(mod_start, mod_end)` | помечает страницы модуля занятыми | иначе PMM выдаст их под другое → tar затрётся до парсинга |
| `heap::HEAP.init(...)` | инициализирует кучу (1 MB) | `Vec` в `ramfs::init` аллоцирует из неё |
| `ramfs::init()` | парсит tar → `FILES` | **строго после** heap и reserve |

После `ramfs::init()` память модуля больше не используется — все байлы уже
скопированы в `Vec`, поэтому ФС полностью writable, а исходный tar остаётся
нетронутым «оригиналом».

## 2. Формат tar (USTAR), который мы читаем

Архив — это последовательность записей по 512 байт. У каждого файла: один блок
**заголовка** + данные, добитые нулями до кратности 512.

Из всего заголовка нам нужны только два поля:

| Смещение | Размер | Поле | Как читаем |
|----------|--------|------|-----------|
| `0` | 100 | `name` (ASCII, NUL-terminated) | копируем в `File.name[100]` |
| `124` | 12 | `size` (**восьмеричное** ASCII) | `parse_octal(base, off+124, 12)` |
| `512` | — | начало данных файла | `base + off + 512` |

Переход к следующему заголовку:

```
off += 512 + ((size + 511) & !511);   // 512 (header) + данные, округлённые вверх до 512
```

Признак конца архива — **пустое имя** (`name[0] == 0`): tar завершается двумя
нулевыми блоками, нам достаточно первого нулевого байта имени.

> Почему размер в восьмеричной? Так устроен исторический формат tar. `parse_octal`
> читает цифры `'0'..'7'` и останавливается на первом не-цифре (NUL/пробел), так
> что ведущие/хвостовые пробелы поля не мешают.

## 3. Структура в памяти

```rust
pub struct File {
    pub name: [u8; 100],   // фиксированное поле, как в tar; хвост забит нулями
    pub data: Vec<u8>,     // содержимое живёт в heap, растёт/сжимается свободно
}

static mut FILES: Vec<File> = Vec::new();
```

- **Имя** — фиксированный массив `[u8; 100]`, а не `String`: длина имени
  вычисляется на лету как позиция первого `0` (`position(|&b| b == 0)`).
  Практический предел имени — **99 байт** (при `create` копируется `min(len, 99)`,
  чтобы гарантированно остался терминирующий ноль).
- **Данные** — `Vec<u8>` в куче. Именно это делает ФС записываемой: `write`
  просто пересобирает вектор, `create` заводит пустой.
- **`FILES`** — глобальный список. Пока однопоточно и без вытеснения в этих
  путях, поэтому обходимся `static mut` + `unsafe` (см. §6, грабли).

## 4. API модуля

| Функция | Сигнатура | Что делает | Возврат |
|---------|-----------|-----------|---------|
| `init` | `fn init()` | парсит tar по `MOD_START`, наполняет `FILES` | — |
| `list` | `fn list(cb: impl FnMut(&[u8], usize))` | вызывает `cb(имя, размер)` на каждый файл | — |
| `find` | `fn find(name) -> Option<&'static [u8]>` | ищет файл, отдаёт срез его данных | `Some/None` |
| `write` | `fn write(name, data) -> bool` | **полностью заменяет** содержимое файла | `true` если найден |
| `create` | `fn create(name) -> bool` | создаёт пустой файл; отказ, если имя занято | `true` если создан |
| `dealeate` | `fn dealeate(name) -> bool` | удаляет файл из `FILES` | `true` если удалён |

Семантика, о которой легко забыть:

- `write` — это **truncate + запись целиком** (`data.clear()` затем
  `extend_from_slice`). Ни дозаписи (append), ни записи по смещению нет.
- `find` возвращает `&'static [u8]` — срез на данные внутри `FILES`. Он валиден,
  пока файл не удалён/перезаписан; вызывающий код (cat) использует его сразу.
- `create` не перезаписывает существующий: сначала проверяет коллизию имени.

## 5. Команды шелла (`src/shell.rs`)

Тонкие обёртки над API. Аргументы — байтовые срезы; `split_first_word` делит
строку на «первое слово» (имя файла) и «остаток» (данные).

| Команда | Обёртка | Делает |
|---------|---------|--------|
| `ls` | `cmd_ls` | `ramfs::list` → печатает `имя размер` построчно |
| `cat <file>` | `cmd_cat` | `ramfs::find` → печатает содержимое или `file not found` |
| `write <file> <text>` | `cmd_write` | `ramfs::write(name, text)` — заменяет содержимое |
| `touch <name>` | `cmd_create` | `ramfs::create(name)` — пустой файл |
| `rm <name>` | `cmd_rm` | `ramfs::dealeate(name)` — удаление |

Пример сессии:

```
> touch note.txt
file created
> write note.txt hello world
written to file
> cat note.txt
hello world
> ls
note.txt 11
hello.txt 18
> rm note.txt
file deleted
```

## 6. Грабли и текущие ограничения

Здесь честно, что уже поймано и что пока хрупко.

1. **Порядок инициализации.** `ramfs::init()` обязан идти **после**
   `heap::HEAP.init()` (иначе `Vec` не из чего аллоцировать) и после
   `pmm::reserve(mod_start, mod_end)` (иначе PMM выдаст страницы tar под кучу и
   архив затрётся ещё до парсинга). Порядок в `main.rs`: PMM → reserve → heap →
   `ramfs::init`.

2. **`static mut FILES` + `unsafe` везде.** Работает только потому, что доступ к
   ФС сейчас однопоточный и не вытесняется в критических местах. При появлении
   конкурентного доступа (несколько процессов, обработчик прерываний) это UB.
   Кандидат на замену — `spin::Mutex<Vec<File>>` или `Once`-обёртка.

3. **`find` отдаёт `&'static`.** Формально это ложь: срез живёт лишь до
   следующего `write`/`rm` этого файла. Пока спасает то, что `cat` печатает
   данные немедленно. Небезопасно хранить этот срез между командами.

4. **Никаких каталогов, прав, времени.** Список плоский. Имя с `/` внутри —
   просто имя, а не путь. Метаданных tar (uid/gid/mode/mtime) не читаем.

5. **Предел имени — 99 байт.** Поле `name` = 100 байт; при `create` берём
   `min(len, 99)`, чтобы остался терминирующий ноль. Длинные имена из tar
   (USTAR prefix / GNU long name) не поддерживаются.

6. **`write` не умеет append.** Каждая запись затирает файл целиком. Для
   редактора/`>>` понадобится отдельный путь (запись по смещению или дозапись).

7. **Опечатка в API: `dealeate`.** Должно быть `delete`/`deallocate`. Косметика,
   но перед защитой стоит переименовать — вылезает в публичном API модуля.

## 7. Карта функций

| Файл | Функция | Роль |
|------|---------|------|
| `main.rs` | разбор тега `3`, `reserve`, `MOD_START/END` | доставляет tar в память |
| `ramfs.rs` | `init` | tar → `FILES` |
| `ramfs.rs` | `list` / `find` | чтение |
| `ramfs.rs` | `create` / `write` / `dealeate` | запись/создание/удаление |
| `shell.rs` | `cmd_ls/cat/write/create/rm` | команды пользователя |
| `shell.rs` | `parse_octal` | восьмеричный размер из tar-заголовка |
| `shell.rs` | `split_first_word` / `trim` | разбор аргументов команд |
