# План развития Tomorrow OS

Полная дорожная карта: от текущего состояния до запуска немодифицированных
Linux-бинарей. Детальный план файловой системы — в `md/plan-fs.md`.

Принцип прежний: маленькие шаги, каждый компилируется и проверяется
отдельно. Треки можно перемежать, но зависимости между ними жёсткие —
они указаны. Это план на месяцы, и это нормально: каждый трек — законченная
тема системного программирования.

---

## Что уже есть (июль 2026)

| Подсистема | Состояние |
|---|---|
| Boot (GRUB MB2, 32→64) | готово; identity map 0–1 ТБ, higher-half — только 2 МБ |
| PMM / VMM | готово (bitmap-страницы; map 4К в свободные PML4-слоты) |
| Heap | free-list с рабочим dealloc; без коалесценции |
| IDT / APIC / HPET / TSS | готово |
| Процессы ring 3 | работают, НО: общий CR3, код = функции ядра, virt-адреса с хаком `pid*0x1000` (`process.rs:56,64`) |
| Планировщик | round-robin, 64 слота; состояний «спит/ждёт» нет |
| Syscalls | write / exit / yield (`syscall.rs:77`); есть зачаток модели безопасности: `syscall_mask`, `domain`, `token` в `Process` |
| Консоль | framebuffer + PSF2; скролла нет — экран очищается целиком |
| Шелл | команды, line discipline (program_output) |
| USB / xHCI | работает на железе: хабы + TT, boot-клавиатура |
| ramfs | writable, `Mutex<Vec<File>>` — переделка по `plan-fs.md` |

Архитектурные долги, из которых вырастает план:
1. нет изоляции процессов (один pml4 на всех);
2. нет настоящего userspace (spawn запускает функции ядра, `shell.rs:258`);
3. higher-half 2 МБ — ядро живёт на identity map (`process.rs:35`);
4. планировщик не умеет блокировать;
5. отладка на железе = фотография замёрзшего экрана.

## Вехи (зачем всё это)

- **М1 «настоящая программа»**: бинарь с диска читает клавиатуру и файлы
  через syscalls (треки A, B, C).
- **М2 «настоящая ОС»**: процессы изолированы, ядро переживает любой
  мусор из userspace (треки D, G).
- **М3 «маленький Unix»**: fork/exec/wait, pipes, `cat file | wc` в шелле
  (трек I).
- **М4 «Linux ABI»**: статический musl hello-world, собранный обычным
  `gcc` на Linux, запускается на Tomorrow без изменений (трек F).
- **М5 «busybox»**: busybox sh работает как шелл Tomorrow (трек F, финал).

## Трек A — Файловая система (в работе)

См. `md/plan-fs.md`: этап 2 (inode-ramfs на `Arc`) → этап 3 (fd-таблица,
sys_open/read/write/close) → этап 4 (коалесценция и пр.).
Фундамент для C (exec читает ELF из ramfs) и I (pipes живут в fd-таблице).

## Трек B — Блокирующий ввод и состояния планировщика

Цель: процесс умеет ждать событие, не сжигая CPU.

1. **B.1** `ProcessState::Blocked(причина)`; schedule() пропускает
   заблокированных. Проверка: Blocked не получает кванты, после Unblock — получает.
2. **B.2** `sys_sleep(ms)`: таймерный тик будит по дедлайну. Проверка:
   два процесса спят разное время, просыпаются в правильном порядке.
3. **B.3** Кольцевой буфер клавиатуры в ядре + `sys_read(fd 0)`: пусто —
   Blocked до нажатия, xHCI-обработчик будит. Проверка: программа читает
   строку; пока ждёт — `ps` показывает Blocked.

После B.3 шелл может стать обычным процессом — красивая веха, не обязательная.

## Трек C — Настоящий userspace (ELF из initrd)

Зависит от: A этап 3; желательно B.1.

1. **C.1** Отдельный crate `user/hello`: no_std, syscalls через asm-обёртки,
   свой linker script. В initrd.tar при сборке (make.sh).
2. **C.2** ELF-парсер в ядре: заголовок + program headers, только PT_LOAD.
   Проверка: команда шелла печатает сегменты файла из initrd.
3. **C.3** `exec <file>`: страницы, копирование сегментов, PAGE_USER, стек,
   entry из ELF. `spawn a|b` уходит. Проверка: два разных бинаря параллельно.
4. **C.4** Передача argv: SysV-раскладка user-стека (argc, argv[], NULL).
   Сделать сразу по ABI System V AMD64 — в треке F это же место расширится
   до envp/auxv, лучше не переучиваться.

## Трек D — Память: honest higher-half и per-process CR3

Самый сложный трек. Делать после C, иначе exec переделывается дважды.
Это он убивает хаки `pid*0x1000`.

1. **D.1** Полный higher-half: всё ядро (код, стеки, куча, DMA-структуры
   xHCI) адресуется через 0xFFFF8000...; 2 МБ расширяются до нужного.
   Закрывает грабли из `process.rs:35`. Проверка: ядро живёт без
   identity-обращений к своим структурам (identity map остаётся для DMA/MMIO).
2. **D.2** Per-process PML4: kernel-половина общая (одни PDPT на всех),
   user-половина своя; CR3 переключается в context switch. Проверка: два
   процесса мапят один virt в разные phys и не видят друг друга.
3. **D.3** #PF из ring 3 убивает процесс (а не ядро): отдельный стек
   через IST уже есть у #DF — оформить и для #PF. Проверка: программа
   пишет по 0x0 → «killed», шелл жив.

## Трек G — Изоляция и безопасность (вперемешку с D)

Цель М2: ядро переживает ЛЮБОЙ мусор из userspace.

1. **G.1** Централизованные `copy_from_user` / `copy_to_user`: единственное
   место, где ядро трогает user-указатели; проверка диапазона и маппинга.
   Все syscalls переводятся на них. Проверка: fuzz-программа зовёт syscalls
   со случайными аргументами — ядро не падает (это же тест G-финала).
2. **G.2** W^X для user-маппингов: код — без PAGE_WRITABLE, данные — c NX
   (бит 63 PTE; включить EFER.NXE в boot.s). Проверка: прыжок в стек → #PF.
3. **G.3** SMEP/SMAP (биты CR4), если CPU умеет: ядро физически не может
   исполнять/читать user-страницы мимо copy_*_user. Дешёвая строчка,
   ловит целый класс багов ядра.
4. **G.4** Оформить свою модель `syscall_mask`/`domain`/`token` (зачаток
   уже в `Process`): документировать в md/, маска проверяется в диспетчере —
   это самодельный seccomp; понадобится осмысленный набор доменов.
5. **G.5** Далее по вкусу: guard page под user-стеком, ASLR (рандомный
   сдвиг базы стека/кучи), stack canary в ядре.

## Трек I — Unix-семантика процессов (М3)

Зависит от: C, D (свой CR3 на процесс — иначе fork невозможен), B.

1. **I.1** `sys_spawn(path)` → полноценный `fork()` + `execve()`: fork
   копирует адресное пространство (для начала честной копией страниц;
   copy-on-write — отдельное упражнение позже), execve заменяет образ.
2. **I.2** `sys_wait()`: родитель Blocked до смерти ребёнка, забирает
   exit code; зомби-состояние между exit и wait. Проверка: шелл ждёт
   завершения программы, а с `&` — не ждёт.
3. **I.3** Pipes: кольцевой буфер в ядре, два fd (read/write конец),
   блокировка на полном/пустом. Шелл учит `|`. Проверка: `cat hello.txt | wc`.
4. **I.4** Сигналы-минимум: SIGKILL (ядро просто убивает) и SIGINT
   (Ctrl+C в line discipline → убить foreground-процесс). Полноценная
   доставка с user-обработчиками — далёкий горизонт.
5. **I.5** stdin/stdout как наследуемые fd 0/1/2 (сейчас fd 1 захардкожен
   в syscall): ребёнок наследует таблицу — после этого редирект `> file`
   в шелле становится тривиальным.

## Трек F — Linux ABI (М4–М5)

Цель: бинарь, собранный на Linux (`gcc -static`, musl), работает без
изменений. Зависит от: I (нужны fork/execve/wait), D, G.
Это НЕ переписывание — это переименование + доделки: семантика к этому
моменту уже юниксовая.

1. **F.1** Конвенция ошибок как в Linux: возврат `-errno` (u64 в диапазоне
   -4095..-1), таблица errno в md/. Свои user-программы переходят на неё.
2. **F.2** Таблица syscall-номеров x86_64 Linux: read=0, write=1, open=2,
   close=3, exit_group=231... Диспетчер `syscall.rs:77` становится таблицей.
   Свои программы пересобираются с новыми номерами — и это всё ещё свои
   программы: удобная промежуточная проверка.
3. **F.3** Что дозагрузить для static-musl hello (выяснится по факту через
   serial-лог «unknown syscall NNN» — сделать такой лог первым делом):
   - стек по ABI: argc/argv/envp/**auxv** (AT_PAGESZ, AT_PHDR/PHENT/PHNUM,
     AT_ENTRY, AT_RANDOM, AT_NULL) — расширение C.4;
   - `arch_prctl(ARCH_SET_FS)` — TLS, musl требует до main;
   - `set_tid_address`, `brk` (или honest `mmap(MAP_ANONYMOUS)`), `writev`,
     `ioctl(TIOCGWINSZ)` можно стабом, `fstat` для stdout.
4. **F.4 = М4**: `hello` из musl-gcc печатает на консоль Tomorrow.
   Дальше — `printf` с форматированием, чтение файлов (openat! musl зовёт
   его, не open), таймеры (clock_gettime — у нас есть HPET).
5. **F.5 = М5**: busybox static: нужны честные fork/execve/wait4, dup2,
   pipe2, сигналы получше, /dev/null-подобные заглушки. Когда busybox sh
   запустится — Tomorrow формально самодостаточна: свой шелл можно удалять.

Замечание про масштаб: между F.4 и F.5 — недели итераций «запустил →
unknown syscall → реализовал → дальше». Это самый затягивающий режим
разработки, serial-лог обязателен.

## Трек E — Инструменты (постоянный фон)

- **E.1 Serial COM1** (~50 строк, порт 0x3F8): boot-лог, паники, «unknown
  syscall» — всё в UART. Железо: USB-TTL кабель; QEMU: `-serial stdio`.
  **Сделать первым из всего плана** — удешевляет каждую следующую отладку.
- **E.2** Скролл консоли: memmove пиксельных строк вместо очистки экрана.
- **E.3** Панике — backtrace: пройти по цепочке RBP (собираем с
  force-frame-pointers), печатать адреса; символы смотреть в `nm` на хосте.
  Опционально: вшить таблицу символов в бинарь и печатать имена.
- **E.4** Хостовые юнит-тесты: `FreeList` и tar-парсер не зависят от
  железа — вынести в модули, тестируемые обычным `cargo test` на хосте
  (heap: выделить буфер через `std`, скормить как регион). Ловит логические
  баги за секунды вместо перезагрузок железа.
- **E.5** Смоук-тест в QEMU одной командой: собрать ISO, загрузить
  headless, снять screendump / прочитать serial, grep «HEAP ok … Scheduler ok».
  Гонять перед каждой прошивкой флешки.
- **E.6** GDB: в QEMU уже сейчас `-s -S` + `target remote :1234` (символы
  из tomorrow.elf). На железе — gdb-stub по serial: амбициозно, по желанию.
- **E.7** Отладочные команды шелла: `ps` с состояниями, `heap`, `lspci`,
  `lsusb`, `mem` (карта PMM).
- **E.8** Тулчейн user-программ: cargo workspace (`kernel` + `user/*`),
  общий crate с номерами syscalls; на этапе F — обычный musl-gcc с Linux.

## Далёкий горизонт (после М5, если не надоест)

- **Диск**: AHCI (SATA) или NVMe → FAT32 read → ОС живёт не в initrd;
  затем запись и свой блочный кэш.
- **Сеть**: virtio-net (QEMU) или e1000 + стек smoltcp → ping, потом сокеты.
- **SMP**: разбудить AP-ядра через ACPI MADT (он уже парсится), per-CPU
  структуры, спинлоки станут по-настоящему нужны.
- **COW fork, mmap файлов, page cache** — «взрослая» виртуальная память.
- **Графика**: со временем — двойная буферизация консоли, окна… (отдельная
  вселенная, трезво оценить желание).
- Полные сигналы, sysfs/procfs-подобие, users/permissions в ФС.
