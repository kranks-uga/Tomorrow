# Шелл и ввод с клавиатуры (`src/shell.rs`, `src/keyboard.rs`)

Документация к пользовательскому интерфейсу Tomorrow OS. Шелл — это **не
отдельный процесс**: у него нет своего контекста и стека. Вся его логика
выполняется в контексте прерывания клавиатуры. Драйверы клавиатуры (PS/2 и USB)
декодируют нажатие в ASCII-байт и зовут единую точку входа `shell::on_char`,
которая копит строку и по Enter запускает команду.

> Два источника ввода сходятся в одной воронке: PS/2 через IRQ1
> (`keyboard_irq_handler`) и USB-HID через опрос в таймерном ISR
> (`xhci::poll_hid`). Оба отдают уже декодированный ASCII в `on_char`, поэтому
> шеллу всё равно, какая клавиатура подключена.

---

## 1. Путь символа: клавиша → экран

```
 PS/2 IRQ1                         USB (за хабом/TT)
 keyboard_irq_handler             xhci::poll_hid  (в таймерном ISR)
   inb(0x60) скан-код               HID boot report
   pc_keyboard → DecodedKey           → HID usage → ASCII
        │                                   │
        └──────────► shell::on_char(ascii) ◄┘
                          │
              накопление в LINE[LEN]  +  эхо на экран
                          │
                     Enter ('\n') → execute() → команда
```

Отдельной задачи-шелла нет → буфер строки статический (`static mut LINE/LEN`),
и это корректно ровно потому, что оба источника — контекст прерывания и не
вытесняются друг другом на этом пути.

---

## 2. Драйвер PS/2 (`keyboard.rs`)

Читает скан-коды из порта `0x60`, декодирует библиотекой `pc_keyboard`
(`ScancodeSet1`, раскладка `Us104Key`), ASCII (`< 128`) отдаёт в `on_char`.

**Главная грабля — edge-triggered IRQ1** (подробно в комментарии обработчика):

- i8042 поднимает IRQ только по фронту OBF (`0x64` бит 0): `0 → 1`. Одно нажатие
  = минимум 2 байта (make+break), расширенные клавиши — плюс префикс `0xE0`.
- Если вычитать **один** байт, OBF остаётся `1`, нового фронта нет → IRQ больше
  не приходят, клавиатура «умирает». Поэтому буфер опустошают в цикле, **пока
  OBF взведён**.
- Предохранитель от залипшей USB-legacy-эмуляции: счётчик итераций. Но порог
  должен быть заведомо больше нормального всплеска — `16` было мало (после
  участка с закрытыми прерываниями в буфере копятся десятки байт, обработчик
  обрезался на 16-м и добивал клавиатуру). Сейчас порог `256`, а при
  срабатывании буфер **всё равно высасывается досуха**, чтобы OBF гарантированно
  ушёл в `0`.
- В конце — `lapic::eoi`.

## 3. Накопитель строки и эхо (`on_char`)

`on_char(ch)` — единая точка приёма декодированного ASCII от любого драйвера:

| Байт | Действие |
|------|----------|
| `'\n'` (Enter) | печать перевода строки → `execute()` → `LEN = 0` → новый `prompt()` |
| `0x08` (Backspace) | если `LEN > 0`: `LEN -= 1`, `console().backspace()` |
| прочий ASCII | если влезает в `BUF_LEN`: дописать в `LINE`, `LEN += 1`, эхо-печать |

Константы: `PROMPT = "User@Tomorrow> "`, `BUF_LEN = 128`. Флаг `READY`
(ставится в `init`) гасит ввод до подъёма шелла — ранний boot-вывод идёт мимо.

## 4. Line discipline: почему `program_output`

Вывод пользовательских процессов (`SYS_WRITE`, fd=1) и интерактивная строка
ввода делят **одну консоль и общий курсор**. Если процесс печатает прямо в
момент набора команды — его символы вклиниваются в набираемую строку, ввод и
вывод перемешиваются.

Решение — `program_output(bytes)`:

```
1. если шелл ещё не READY → пишем напрямую (ранний boot)
2. сохранить RFLAGS, cli               ← атомарность относительно IRQ/poll_hid
3. backspace × (PROMPT_LEN + LEN)      ← стереть видимое приглашение + буфер
4. write_bytes(bytes)                  ← вывод программы на «чистую» строку
5. redraw_line()                       ← перерисовать приглашение + буфер
6. popfq                               ← вернуть прежний IF (лишних IRQ не включит)
```

Строка ввода всегда остаётся целой внизу. `cli`/`popfq` (а не `sti`) — чтобы не
включить прерывания там, где вызвали из контекста с закрытым IF. `SYS_WRITE`
уже маскирует IF, так что `on_char` и `poll_hid` не вклинятся между стиранием и
перерисовкой.

## 5. Диспетчер команд (`execute`)

`execute()` берёт `LINE[..LEN]`, тримит, делит на первое слово (команда) и
остаток (аргументы) через `split_first_word`, затем — цепочка `if eq(cmd, ...)`.

| Команда | Обработчик | Действие |
|---------|-----------|----------|
| `help` | `cmd_help` | список команд |
| `clear` / `cls` | — | `console().clear()` |
| `echo <text>` | — | печать аргументов |
| `ticks` | — | счётчик тиков таймера |
| `ps` | — | список процессов |
| `spawn <a\|b>` | — | создать демо-процесс |
| `kill <pid>` | — | завершить процесс |
| `mem` | — | свободная физ. память (PMM) |
| `reboot` | — | перезагрузка |
| `shutdown` / `poweroff` | `cmd_shutdown` | ACPI S5 (запись в PM1x_CNT) |
| `module` | `mods` | `MOD_START` / `MOD_END` |
| `ls` / `cat` / `write` / `touch` / `rm` | `cmd_*` | файловая система (см. `ramfs.md`) |

Неизвестная команда → `unknown command: <cmd>`.

## 6. Грабли и заметки

1. **Шелл живёт в контексте прерывания.** Долгая команда блокирует обработку
   ввода. Пока команды короткие — приемлемо, но это причина, по которой буфер
   статический и почему `program_output` синхронизируется через `cli`.

2. **Рассинхрон usage-строк.** В `cmd_help` для `write` не показан `<text>`, а
   `cmd_create`/`cmd_rm` печатают usage со словом «create» для команды `touch`.
   Косметика, но заметно на защите — стоит согласовать с реальными именами.

3. **`cmd_create`/`cmd_rm` не выходят после usage.** При пустых аргументах
   печатают подсказку, но затем всё равно идут в `split_first_word` пустой
   строки (имя окажется пустым). Безвредно, но логичнее `return` после usage.

4. **`ScancodeSet1` + `Us104Key` захардкожены.** Другая раскладка/скан-сет —
   отдельная работа; сейчас цель — рабочий ASCII-ввод, а не полноценная
   i18n-клавиатура.
