## Этап 2 — inode-ramfs

**Цель:** файл живёт сам по себе (`Arc<Inode>`), каталог — лишь таблица
«имя → ссылка». Удаление из каталога не убивает файл, пока на него есть
ссылки — та самая unlink-семантика Unix, без которой невозможен этап 3.

**Чем плох текущий `Mutex<Vec<File>>`:**
- индекс файла в `Vec` нестабилен (`remove` сдвигает, рост перемещает) —
  «держать файл открытым» нельзя;
- любой доступ запирает весь ramfs (`with_file` работает под глобальным локом).

### 2.1 Структуры

```rust
use alloc::{string::String, sync::Arc, vec::Vec};
use spin::Mutex;

pub struct Inode {
    pub data: Mutex<Vec<u8>>,   // лок на ОДИН файл, а не на всю ФС
}

struct DirEntry {
    name: String,               // не [u8; 100]: куча теперь настоящая
    inode: Arc<Inode>,
}

static ROOT: Mutex<Vec<DirEntry>> = Mutex::new(Vec::new());
```

`Arc` — из `alloc::sync`, счётчик атомарный, на x86_64 работает без ОС.
Каждый `Arc::clone`/drop — это инкремент/декремент и, в конце, `dealloc`:
живёт на аллокаторе из этапа 1.

**Готово, когда:** компилируется рядом со старым кодом (как делали с кучей).

### 2.2 `init()` — перенос tar-парсера

Парсер initrd.tar переезжает почти без изменений: вместо `File` собирается
`Arc::new(Inode { ... })` + `DirEntry`. Имя — `String::from_utf8_lossy(...)`
или ручная сборка из байтов до NUL.

**Готово, когда:** `ls` (пока на старом API) заменён и показывает те же файлы.

### 2.3 Новый API

```rust
pub fn lookup(name: &str) -> Option<Arc<Inode>>;      // вместо with_file
pub fn list(cb: impl FnMut(&str, usize));
pub fn create(name: &str) -> Option<Arc<Inode>>;      // None = уже существует
pub fn unlink(name: &str) -> bool;                    // бывший delete
```

`lookup` — ключевая перемена: он **клонирует `Arc` и отпускает лок ROOT**.
Дальше вызывающий работает с `inode.data.lock()` — другие файлы свободны.

**Ловушка — порядок локов:** никогда не брать `inode.data.lock()`, держа
`ROOT.lock()`. Правило: ROOT лочится только на время поиска/вставки/удаления
записи; с данными работаем уже после отпускания. Нарушение = дедлок, который
на железе выглядит как «замёрзший экран» (см. заметку про скрытые паники —
тут будет честный хэнг, даже без паники).

**Готово, когда:** `cat`, `write`, `touch`, `rm` в шелле переведены на новый
API, старые `File`/`FILES`/`search_name` удалены.

### 2.4 Проверка на железе

1. `ls`, `cat hello.txt` — initrd читается;
2. `touch a` → `write a ...` → `cat a` → `rm a` → `ls`;
3. стресс: `write a <длинная строка>` раз 30–50 подряд — куча не течёт
   (раньше каждый `write` терял старый `Vec`).

### 2.5 (опционально) команда `heap` в шелле

Обход free-list с суммой свободных байтов и числом блоков:

```
User@Tomorrow> heap
free: 987136 bytes in 3 blocks
```

Десять строк кода, а даёт глаза: видно и утечки (free тает), и фрагментацию
(блоков всё больше) — пригодится в этапе 4.

## Этап 3 — файловые syscalls

**Цель:** userspace-программа открывает файл, читает/пишет по частям,
закрывает. Ядро отслеживает открытые файлы per-process.

### 3.1 fd-таблица в `Process`

```rust
pub struct OpenFile {
    pub inode: Arc<Inode>,
    pub offset: usize,          // позиция чтения/записи
}

// в Process:
pub fds: Vec<Option<OpenFile>>,   // индекс = fd
```

Конвенция: fd 0/1/2 зарезервированы (консоль), файлы — с 3.
`sys_open` ищет первый `None` начиная с 3 (или пушит).

**Вопрос на подумать:** почему `offset` лежит в fd-таблице, а не в `Inode`?
(Подсказка: два процесса читают один файл.)

### 3.2 `sys_open(path_ptr, path_len) -> fd | ошибка`

- скопировать имя из user-памяти, **проверив указатель+длину** (диапазон
  user-страниц процесса; ядро не должно падать от мусорного указателя);
- `ramfs::lookup` → `OpenFile { inode, offset: 0 }` → свободный слот → fd.

Ошибки — в стиле существующего диспетчера (`u64::MAX`, `MAX-1`, ...
или отрицательные как в Linux; выбрать одну конвенцию и записать в md/).

### 3.3 `sys_read` / `sys_write` по fd

- `sys_read(fd, buf, len)`: скопировать из `inode.data` начиная с `offset`,
  вернуть фактическое число байт (0 = EOF), сдвинуть `offset`;
- `sys_write(fd, buf, len)`: fd 1 — как сейчас, через
  `shell::program_output` (line discipline консоли); fd ≥ 3 — запись в
  `inode.data` по `offset` (перезапись + расширение `Vec` при выходе за конец).

### 3.4 `sys_close(fd)`

`fds[fd] = None` — drop `OpenFile` → drop `Arc`. Если файл уже unlink-нут
и это была последняя ссылка — здесь память файла и освободится.

### 3.5 Проверка

Тест-программа в initrd: `open("hello.txt")` → read кусками по 16 байт →
вывод на fd 1 → close. Затем главный тест семантики: открыть файл, `rm` его
из шелла, дочитать из открытого fd — данные живы, после close исчезают.

## Этап 4 — отложенное (по желанию, после этапа 3)

- **Коалесценция free-list:** держать список сортированным по адресу и
  сливать соседей при `add_free_region`. Команда `heap` из 2.5 покажет
  эффект наглядно.
- Флаги `open` (O_TRUNC / O_APPEND / O_CREAT), `sys_seek`.
- Каталоги и пути (`/bin/hello`) — понадобится, когда файлов станет много.
